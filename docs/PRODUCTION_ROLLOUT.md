# Guarded Production Rollout Executor

## B3-C2A boundary

B3-C2A develops and rehearses the Production rollout executor. It does not execute a Production rollout, deploy Production or Staging, create a Production data plane, create a Qwen Secret, or authorize B3-C2B. C1 requirements, C2 Scope comparison, D Action/Risk/Weekly Report, Rerank, ANN, OCR, Tool Calling, and Agent Execution remain frozen.

Formal Production rollout is a later B3-C2B operation against an independently reviewed and merged main commit. The default state is fail-closed: no formal Authorization exists, `productionRolloutAuthorized` remains false in B3-C1 Go/No-Go evidence, and Production `--apply` without an independently signed Authorization exits with `PRODUCTION_APPLY_NOT_AUTHORIZED` and code 78.

## Authorization

The versioned schema is `release/production-rollout-authorization.schema.json`. The producer is `projectai-release-tool`, version `b3-c2-v2`. A formal Authorization is bound to one Release Session, Candidate SHA, App image digest, db-tools image digest, Production baseline digest, Go/No-Go digest, expiry, exactly one Phase, and exactly one action: `apply`, `resume`, `rollback`, `finalize`, `lock-clear`, or `image-transfer`.

Formal `production-approval` records require an Ed25519 signature verified with an operator-controlled public key. B3-C2A cannot generate a formal Authorization. Unit tests and isolated rehearsals use only short-lived `synthetic-test` Authorization and ephemeral keys. Recomputing a JSON Digest or writing `productionRolloutAuthorized=true` cannot forge the signature.

Production never accepts a caller-selected key. The only key path is `/srv/projectai/authorization/production-rollout-public-key.pem`; it must be a root-owned, non-symlink Ed25519 public-key file with mode `0400` or `0444`, and its SPKI-DER SHA-256 must match `release/production-rollout-trust.json`. The reviewed B3-C2A fingerprint is `sha256:957296b2fd6b9c4a744fd113f851e6de65f556789d6a0351fd2e6a258e903438`; its private key was never persisted, so it cannot authorize a rollout. B3-C2B requires a separate reviewed key ceremony and Trust Contract update before any formal Authorization can exist.

Authorization lifetime is at most 60 minutes and one Authorization authorizes exactly one Phase/action pair. `/srv/projectai/authorization/rollout-enabled.json` must be a matching root-owned Marker carrying the same Phase and action. Before any rollout mutation, consumption writes and `fsync`s a complete per-Authorization record, publishes it with a cross-process atomic no-replace hard-link claim, removes the private link, and `fsync`s the protected claim directory. The immutable claim is the replay authority: exactly one process can publish it, and every concurrent or later invocation for that signed Authorization returns `PRODUCTION_AUTHORIZATION_REPLAYED` with `already_consumed`. The executor then serializes `used-authorizations.jsonl`, opens it with `O_NOFOLLOW`, verifies regular-file/root-owner/`0600`/single-link metadata, validates the full Digest chain and unique IDs, performs one `O_APPEND` write, and `fsync`s both file and parent directory. If claim publication succeeded but Journal append was interrupted, the next replay first validates and reconciles the one missing Digest-linked record, then returns `already_consumed`. A dead-PID, malformed, live, or otherwise ambiguous internal mutex is never auto-cleared or overwritten; it fails closed as unknown state and requires explicit review. Unsafe Journal metadata, a malformed claim, broken chain, partial record, or uncertain append follows the same rule.

## Phase execution

Every Phase is a separate command and produces a separate digest-pinned report:

```bash
npm run production:phase -- --phase=0 --environment=production --dry-run ...
npm run production:phase -- --phase=0 --environment=production --apply ...
npm run production:verify -- --phase=0 --environment=production ...
```

There is no command that automatically promotes Phase 0 through Phase 6. Apply independently collects a fresh live Inventory, consumes its one-Phase Authorization, performs only the mutation, and stops in `awaiting_verification`; its command-result Digest, mutation window, and Journal event are bound to the Apply Report. `production:verify` rejects caller-written Verification JSON, collects a new live Inventory and a producer-bound Observation Report, and verifies the Session, Phase, Candidate SHA, Manifest/App/db-tools Image Digests, Apply/command Digests, observation window, and Journal binding before it alone may write `succeeded`. Phase 1–6 require the previous generated verification report and matching post-state Digest.

The fixed sequence is:

1. Phase 0 — baseline, deployment/migration locks, capacity, Nginx/Compose, public HTTP, configuration backup, and applicable data backup.
2. Phase 1 — private PostgreSQL 17/pgvector 0.8.1 and MinIO bootstrap, private bucket initialization, and controlled migrations.
3. Phase 2 — immutable App and Document Worker with Assistant/Embedding disabled and lexical retrieval.
4. Phase 3 — operator-prepared Qwen Secret metadata verification and Assistant lexical enablement.
5. Phase 4 — Embedding Worker, bounded first batch of at most 100 Chunks, and budget/unknown checks.
6. Phase 5 — Shadow retrieval with lexical Prompt evidence and at least 30 controlled requests or the reviewed observation window.
7. Phase 6 — Hybrid retrieval with citation, fallback, isolation, archive/current-version, latency, and cost verification.

The formal observation defaults are never zero: Phase 2 is 15 minutes, Phase 3 is 30 minutes, Phase 4 is the first 100 Chunks plus 30 minutes, and Phase 5/6 require at least 30 controlled requests and 30 minutes. Rehearsal may shorten elapsed time while testing the same non-zero gate logic.

## Lock and journal

The fixed Production lock is `/srv/projectai/.production-rollout-lock`. A complete `0600` regular file is fsynced and published with a no-replace hard-link claim, so concurrent acquire has exactly one winner. A publication interrupted after the link is still recognized as the same complete owner-only record instead of being treated as corrupt. Acquire, update, release, and stale-clear share one fixed `0700` lifecycle-guard directory. The directory is only a container: guard ownership begins when a complete, pre-fsynced, Digest-bound `0600 metadata.json` is atomically hard-linked into it with no replacement. An empty directory left by a crash carries no ownership and can be reused by the next claimant; concurrent publishers race the metadata hard-link and exactly one wins. An interrupted post-link publication with two hard links remains readable. A non-empty container without the exact metadata entry, or symlinked, wrong-owner, broad-mode, partial, or otherwise corrupt metadata, fails closed and is never overwritten. A valid guard is never deleted by pathname: completion or reviewed stale takeover atomically renames the exact directory to a deterministic `guardId + Digest` receipt that remains non-empty and is never reused or removed. This prevents a delayed old clearer from moving a newer guard or deleting a reacquired Lock. The digest-protected Lock record binds Lock ID, Session, Candidate SHA, Authorization ID, Phase, owner PID/hostname/UID, Lease Token, heartbeat, original start, and expiry. Every claim or heartbeat performs an exact compare against Session/Lock ID/Lease/owner/Digest and rotates the Lease Token; lease, heartbeat, stale, and clear decisions sample time only after the guarded record has been read, so time spent waiting for a lifecycle guard cannot revive an expired lease. Long-running commands heartbeat every 30 seconds and a two-minute active-heartbeat gap fails stale even before the one-hour lease expiry. Release requires the exact current record and active owner and holds the shared guard through Lock removal plus directory fsync.

Expired, heartbeat-timed-out, and dead-PID Locks return `PRODUCTION_DEPLOYMENT_LOCK_STALE`; neither the Lock nor a live lifecycle guard is auto-cleared. `production:lock:review` is read-only and reports both records without exposing the Lease Token. `production:lock:clear` requires explicit Lock ID acknowledgement plus a new pinned `lock-clear` Authorization/Marker, verifies that the local owner is not live, writes a digest-bound clear approval, then removes only the exact reviewed Lock or retires an orphaned stale guard. The Journal records `lock-clear-approved` and `lock-cleared`; a crash between clear and replacement acquire can resume from that consumed approval. If replacement acquire committed before its Journal event, the next mutating command reconciles the different bound Lock ID and consumes the delayed approval before proceeding. Phase, Verify, Rollback, and Finalize all support this recovery path.

The append-only Journal is `/srv/projectai/releases/<release-session-id>/journal.jsonl`. It must remain a `0600`, owner-only, single-link, non-symlink regular file. Each event carries the previous Digest and its own Digest. Every executor decision binds its reviewed tail as `expectedPreviousDigest`; the append is rejected if the live tail differs. Before append, the executor publishes a permanent complete claim keyed by that previous Digest and bound to the exact event, owner PID/hostname/UID, and claim time. A live, remote, malformed, or otherwise unverifiable unmatched claim fails closed. If a same-host/same-UID owner is proven dead and its claim has exceeded the heartbeat timeout, one process may reconcile only that immutable claimed event under the same atomic metadata hard-link recovery-guard protocol; recovery re-samples wall-clock time on every wait iteration, and completion or stale recovery-guard takeover renames the exact guard to a permanent Digest receipt. This closes the claim-before-append crash window without deleting claims, forking the chain, or duplicating the recovered event. If another event was committed or recovered, the pending caller receives `PRODUCTION_ROLLOUT_STATE_CHANGED` and must reload all state and gates before a new append; it may not transparently attach a decision derived from the old tail. Both Finalize reacquire paths then CAS-claim and release only the exact idle Lock they just created. An ambiguous `PRODUCTION_ROLLOUT_STATE_UNKNOWN` preserves that exact Lock for subsequent Journal/Lock reconciliation because an immutable append claim may still commit; deleting it would create a Journal entry bound to a nonexistent Lock. History, claims, and receipts are never rewritten. The allowed flow includes `running → awaiting_verification → succeeded` and `awaiting_rollback_verification → rolled_back`; invalid transitions fail closed.

`npm run production:status` is read-only and reports only sanitized runtime identity, Phase state, lock metadata, latest report Digests, active counts, feature modes, and rollback availability. `npm run production:resume` reconstructs state from the Lock, Journal, Session, Manifest, Inventory, current runtime, and prior Phase report. Ambiguity returns `PRODUCTION_ROLLOUT_STATE_UNKNOWN`.

## Compose and Secret boundary

`docker-compose.production-rollout.yml` is the reviewed base contract. The internal network contains the data plane and all application services. A separate ordinary Egress bridge connects only App and Embedding Worker, allowing Qwen access without giving PostgreSQL, MinIO, MinIO Init, Migration, Storage Operations, or Document Worker Provider egress. Verification reads `docker network inspect projectai-production-egress` and requires exactly the Phase-appropriate App/Embedding Worker membership; any forbidden or unknown member fails closed. PostgreSQL and MinIO publish no host ports; App alone binds `127.0.0.1:3100`.

The App receives database, auth, object-storage application credentials, and Phase-appropriate AI configuration. The Document Worker additionally reads `.env.embedding-production` so successful new Chunks enqueue jobs only while Embedding is enabled, but it never receives Qwen. Phase 4 enables the flag, rebuilds App and Document Worker, then starts the Qwen-mounted Embedding Worker. Independent verification uploads a bounded fictional document and requires Document Worker ingestion, automatic Embedding Job creation, and stored vectors. The first backfill is capped by the cumulative number of missing Chunks, not Versions or Documents; its database-observed Chunk count must match the mutation result and remain at most 100. Rollback stops the Embedding Worker, disables the flag in both App and Document Worker, and preserves existing Jobs, Usage, and vectors.

The Qwen Secret is prepared independently in B3-C2B. The executor checks only existence, regular-file/non-symlink status, owner/group identifiers, permission mode, non-empty size, and mount scope; it does not print or copy the content. Phase 0–2 do not require the Qwen Secret. Phase 3 without safe metadata stops with `PRODUCTION_QWEN_SECRET_REQUIRED`.

## Image and switch contract

Release images are built locally from the exact reviewed Git SHA for `linux/amd64`, saved and checksummed, transferred without floating tags, loaded on the server, and re-inspected. Every Production Apply forces `/srv/projectai` as CWD, uses absolute Compose paths, rejects caller image environment overrides, sets App/db-tools references from the trusted Manifest, and inspects both loaded images for Digest, OS, architecture, revision, and Production label before mutation. Forward Verify checks the actual App and active Worker container metadata; rollback loads its image only from the trusted baseline Manifest and separately verifies the restored runtime.

`npm run production:image -- plan|build|verify|transfer` implements this contract. Transfer Apply independently requires a signed Phase 0 `image-transfer` Authorization, both archive SHA-256 values, and `PROJECTAI_PRODUCTION_IMAGE_TRANSFER_ENABLED=1`. It verifies the fixed root-owned, non-symlink claim helper plus a Digest-pinned dependency bundle before any local claim modules load. It accepts only a structured receipt matching Authorization ID, Phase, Action and Journal entry Digest; only then may it create the remote directory, run `scp`, or invoke `docker load`. A failed transfer burns that Authorization and requires a new one. After transfer it verifies both checksums and re-inspects both loaded images for Digest, OS, architecture, revision, and Production label. B3-C2A never enables it.

Phase 2 does not run `docker compose down`. Apply starts the private data plane, loads the immutable App, verifies the old baseline has not drifted, replaces only the App/Document Worker, and stops in `awaiting_verification`. The separate Verify command then checks local health, public HTTP, routes, actual runtime images, and restart state. The old image remains pinned in the trusted baseline Manifest for an explicitly authorized rollback.

## Rollback

- Phase 0 retains its owned lock through rollback mutation and rollback verification; only the successful rollback verification releases that exact Lock and retains reports/backups.
- Phase 1 stops only the newly introduced data services and preserves volumes for investigation; the old public App remains untouched.
- Phase 2 restores the exact trusted-baseline App image/runtime, stops both new Workers, and verifies the restored image metadata and HTTP separately.
- Phase 3 disables Assistant and removes the App Qwen mount on rebuild.
- Phase 4 disables Embedding, stops the Embedding Worker, and preserves vectors.
- Phase 6 returns retrieval to Shadow; only the subsequent verified Phase 5 rollback returns it to Lexical.

Only the latest non-rolled-back Phase may be rolled back, enforcing `6 → 5 → 4 → 3 → 2 → 1 → 0`. Rollback Apply uses only the fixed Session state directory and trusted baseline/previous verified Phase target, then stops in `awaiting_rollback_verification`. A distinct rollback Observation Report and fresh rollback Inventory check HTTP, routes, flags, workers, exact App/Worker image state, aggregate business rows, object counts, vectors, active work, restart state, and the previous Phase deployment-state Digest before writing `rolled_back`. A failed Verify retains recovery metadata and may retry verification without repeating the mutation. Earlier Phases cannot be rolled back until higher Phases are individually rolled back and verified.

`production:finalize` first requires a valid Phase 6 Deployment Lock and all seven durable Phase verification reports. Each report must have a valid payload Digest, exact Session/SHA/App/db-tools Image bindings, and a bound `verified` Journal event carrying the same report, post-Inventory, and post-state Digests; only state-preserving Lock/finalization recovery events may follow that verification. The executor then consumes a new Phase 6 `finalize` Authorization, atomically claims the exact Lock, recollects fresh live Inventory, and re-runs the gate. Finalize additionally requires the Manifest App image and commit, both Manifest Worker images, healthy/zero-restart runtime, the Phase 6 state Digest, and exact zero values for all seven Job/Batch/Provider Call/Retrieval Run/Query Call/Execution counters. It writes Final Inventory, report evidence, and `finalization-prepared`, releases the exact Lock, and only then appends `release-completed`. Recovery is never inferred merely from “no Lock”: it requires `--recover-finalization=true`, exact old Lock ID acknowledgement, a new `finalize` Authorization, a prepared record older than the heartbeat timeout, and a same-host/same-UID owner PID proven dead. Recovery then reacquires, recollects live state, and repeats the full gate; a live or unverifiable owner fails closed.

## Rehearsal and evidence

`npm run production:lock:rehearsal` runs the minimal Docker Lock check independently; `npm run production:rehearsal` includes the same check alongside isolated internal and Egress networks, an isolated volume, Fake Qwen, fictional state, and ephemeral keys. Real network probes prove App/Embedding Worker can reach Fake Qwen while database/MinIO probes cannot resolve it. The network-disabled, read-only-root Lock probe starts two containers against the same isolated named volume. The host opens a two-stage `ready → go → armed → attempt` barrier only after both contenders reach each checkpoint, requires exactly one acquire and one `PRODUCTION_DEPLOYMENT_LOCK_HELD`, then verifies that the original winner released the exact Lock and no active lifecycle guard or temporary publication file remains. The command succeeds only after `docker compose down --volumes --remove-orphans` completes and label-based inspection finds zero remaining Compose containers, volumes, or networks. The full rehearsal exercises Apply→Verify for Phase 0–6, Finalize, Resume, rollback mutation/verification, replay/Marker/Trust/Lock gates, and verified cleanup.

CI publishes the following sanitized reports with file and payload Digests: `production-authorization-contract`, `production-phase-state-machine`, `production-rollout-rehearsal`, `production-rollout-rollback`, `production-rollout-resume`, `production-compose-contract`, and `production-secret-boundary`. Reports never contain Secret values, full environment variables, database contents, object keys, user data, Prompt/questions, vectors, or Provider payloads.
